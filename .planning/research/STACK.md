## Supabase Patterns for EdTech Multi-Role Auth

### Authentication Architecture

**Recommended Pattern: Custom Claims via Auth Hooks**

Supabase Auth supports [custom JWT claims](https://supabase.com/docs/guides/database/postgres/custom-claims-and-role-based-access-control-rbac) injected at token generation time. This is the correct pattern for student/parent/teacher roles.

```sql
-- Step 1: Create role enum
CREATE TYPE public.user_role AS ENUM ('student', 'parent', 'teacher', 'school_admin');

-- Step 2: Create user_roles table
CREATE TABLE public.user_roles (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id UUID REFERENCES auth.users ON DELETE CASCADE NOT NULL,
  role user_role NOT NULL,
  school_id UUID REFERENCES public.schools(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (user_id, role, school_id)
);

-- Step 3: Create Auth Hook to inject role into JWT
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event JSONB)
RETURNS JSONB
LANGUAGE plpgsql
STABLE
AS $$
DECLARE
  claims JSONB;
  user_role public.user_role;
  user_school_id UUID;
BEGIN
  -- Get the user's role and school
  SELECT role, school_id INTO user_role, user_school_id
  FROM public.user_roles
  WHERE user_id = (event->>'user_id')::UUID
  LIMIT 1;

  claims := event->'claims';

  IF user_role IS NOT NULL THEN
    claims := jsonb_set(claims, '{user_role}', to_jsonb(user_role));
    claims := jsonb_set(claims, '{school_id}', to_jsonb(user_school_id));
  ELSE
    claims := jsonb_set(claims, '{user_role}', '"pending"');
  END IF;

  event := jsonb_set(event, '{claims}', claims);
  RETURN event;
END;
$$;
```

**Configuration:** Enable this hook in Supabase Dashboard > Authentication > Hooks > Customize Access Token

**Confidence:** HIGH - [Official Supabase documentation](https://supabase.com/docs/guides/database/postgres/custom-claims-and-role-based-access-control-rbac) provides this exact pattern.

### Parent-Student Linking for COPPA

```sql
-- Parent-child relationship table
CREATE TABLE public.parent_student_links (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  parent_id UUID REFERENCES auth.users(id) NOT NULL,
  student_id UUID REFERENCES auth.users(id) NOT NULL,
  consent_given_at TIMESTAMPTZ,
  consent_method TEXT, -- 'email_verification', 'signed_form', etc.
  consent_revoked_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (parent_id, student_id)
);

-- RLS: Parents can only see their linked students
CREATE POLICY "Parents see linked students"
ON public.students FOR SELECT
TO authenticated
USING (
  (SELECT auth.jwt()->>'user_role') = 'parent'
  AND id IN (
    SELECT student_id FROM public.parent_student_links
    WHERE parent_id = auth.uid()
    AND consent_given_at IS NOT NULL
    AND consent_revoked_at IS NULL
  )
);
```

**Confidence:** MEDIUM - Pattern derived from [Supabase RBAC docs](https://supabase.com/docs/guides/database/postgres/custom-claims-and-role-based-access-control-rbac) and COPPA requirements, but no specific EdTech examples found.

## Row Level Security (RLS) Patterns for Education

### Core RLS Strategy

Based on [Supabase RLS documentation](https://supabase.com/docs/guides/database/postgres/row-level-security), use the **JWT claims + school_id tenant isolation** pattern.

```sql
-- Enable RLS on all tables
ALTER TABLE public.students ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.assignments ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.progress_records ENABLE ROW LEVEL SECURITY;

-- Authorization helper function (cache JWT lookup)
CREATE OR REPLACE FUNCTION public.get_user_role()
RETURNS TEXT AS $$
  SELECT (auth.jwt()->>'user_role')::TEXT;
$$ LANGUAGE SQL STABLE SECURITY DEFINER;

CREATE OR REPLACE FUNCTION public.get_user_school_id()
RETURNS UUID AS $$
  SELECT (auth.jwt()->>'school_id')::UUID;
$$ LANGUAGE SQL STABLE SECURITY DEFINER;
```

### Education-Specific RLS Policies

**Students see only their own data:**
```sql
CREATE POLICY "Students see own progress"
ON public.progress_records FOR SELECT
TO authenticated
USING (
  (SELECT public.get_user_role()) = 'student'
  AND student_id = auth.uid()
);
```

**Teachers see students in their school:**
```sql
CREATE POLICY "Teachers see school students"
ON public.students FOR SELECT
TO authenticated
USING (
  (SELECT public.get_user_role()) = 'teacher'
  AND school_id = (SELECT public.get_user_school_id())
);

-- Teachers can update assignments for their students
CREATE POLICY "Teachers manage assignments"
ON public.assignments FOR ALL
TO authenticated
USING (
  (SELECT public.get_user_role()) = 'teacher'
  AND school_id = (SELECT public.get_user_school_id())
)
WITH CHECK (
  (SELECT public.get_user_role()) = 'teacher'
  AND school_id = (SELECT public.get_user_school_id())
);
```

**Parents see only their linked children:**
```sql
CREATE POLICY "Parents see linked children progress"
ON public.progress_records FOR SELECT
TO authenticated
USING (
  (SELECT public.get_user_role()) = 'parent'
  AND student_id IN (
    SELECT student_id FROM public.parent_student_links
    WHERE parent_id = auth.uid()
    AND consent_given_at IS NOT NULL
  )
);
```

### Performance Optimization

From [Supabase RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security):

```sql
-- ALWAYS wrap auth functions in SELECT for caching (94% performance improvement)
-- BAD:  USING (auth.uid() = user_id)
-- GOOD: USING ((SELECT auth.uid()) = user_id)

-- Index all columns used in RLS policies
CREATE INDEX idx_students_school_id ON public.students(school_id);
CREATE INDEX idx_progress_student_id ON public.progress_records(student_id);
CREATE INDEX idx_parent_links_parent ON public.parent_student_links(parent_id);
CREATE INDEX idx_parent_links_student ON public.parent_student_links(student_id);
```

**Confidence:** HIGH - Direct from [Supabase documentation](https://supabase.com/docs/guides/database/postgres/row-level-security) with verified performance patterns.

## Supabase Storage Patterns

### Private Student Content

Based on [Supabase Storage Access Control](https://supabase.com/docs/guides/storage/security/access-control):

```sql
-- Create private bucket for student work
-- (Do this in Supabase Dashboard or via API)

-- Storage RLS: Students access only their own files
CREATE POLICY "Students access own files"
ON storage.objects FOR ALL
TO authenticated
USING (
  bucket_id = 'student-work'
  AND (storage.foldername(name))[1] = (SELECT auth.uid()::text)
)
WITH CHECK (
  bucket_id = 'student-work'
  AND (storage.foldername(name))[1] = (SELECT auth.uid()::text)
);

-- Teachers access files for students in their school
CREATE POLICY "Teachers access school student files"
ON storage.objects FOR SELECT
TO authenticated
USING (
  bucket_id = 'student-work'
  AND (SELECT public.get_user_role()) = 'teacher'
  AND (storage.foldername(name))[1] IN (
    SELECT id::text FROM public.students
    WHERE school_id = (SELECT public.get_user_school_id())
  )
);
```

**File path convention:** `{bucket}/{student_id}/{assignment_id}/{filename}`

**Confidence:** HIGH - Direct from [Supabase Storage documentation](https://supabase.com/docs/guides/storage/security/access-control).

**Confidence:** HIGH - Architectural decision based on scope, not technical limitation.

## COPPA/FERPA Compliance Architecture

### Key 2025-2026 Requirements

From [COPPA amendments (effective June 2025)](https://www.loeb.com/en/insights/publications/2025/05/childrens-online-privacy-in-2025-the-amended-coppa-rule) and [FERPA 2026 checklist](https://www.brightdefense.com/blog/ferpa-compliance-checklist/):

| Requirement | Implementation |
|-------------|----------------|
| Verifiable parental consent | Parent email verification + consent form before student data access |
| Data minimization | Collect only what's needed, no indefinite retention |
| Encryption at rest | AES-256 via defuse/php-encryption for PII |
| Access audit logs | Monolog with 3-year retention |
| Role-based access | Supabase RLS + JWT custom claims |
| Data export | Parent/school can request data export |
| Data deletion | Hard delete capability (COPPA right to deletion) |

### PHP Architecture for Compliance

```
/app
  /api
    /auth         # Handle Supabase auth callbacks, consent flows
    /students     # Student CRUD with audit logging
    /progress     # Learning progress (encrypted PII)
  /middleware
    auth.php      # Validate Supabase JWT, extract role
    audit.php     # Log all data access
    consent.php   # Verify parental consent for minors
  /services
    EncryptionService.php  # Wrap defuse/php-encryption
    AuditService.php       # Wrap Monolog
    SupabaseService.php    # Thin wrapper for supabase-client
```

**Consent Flow (COPPA):**
1. Parent creates account (email verified via Supabase Magic Link)
2. Parent registers student (student has no direct login for demo)
3. Consent form presented, parent must acknowledge
4. Consent timestamp stored, student account activated
5. Parent can revoke consent at any time

**Confidence:** MEDIUM - Architecture pattern derived from [FERPA best practices](https://www.hireplicity.com/blog/ferpa-compliance-checklist-2025) and Supabase capabilities. No specific PHP+Supabase COPPA reference found.

### Database Audit Trail

Use [Supabase's supa_audit pattern](https://supabase.com/blog/postgres-audit) (note: original extension archived Feb 2025, but pattern remains valid):

```sql
-- Create audit schema
CREATE SCHEMA IF NOT EXISTS audit;

-- Audit table (simplified from supa_audit)
CREATE TABLE audit.record_version (
  id BIGSERIAL PRIMARY KEY,
  record_id UUID,
  op VARCHAR(8) NOT NULL,
  ts TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  table_schema NAME NOT NULL,
  table_name NAME NOT NULL,
  record JSONB,
  old_record JSONB,
  user_id UUID,
  user_role TEXT
);

-- Trigger function
CREATE OR REPLACE FUNCTION audit.log_changes()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  INSERT INTO audit.record_version(
    record_id, op, table_schema, table_name,
    record, old_record, user_id, user_role
  ) VALUES (
    COALESCE(NEW.id, OLD.id),
    TG_OP,
    TG_TABLE_SCHEMA,
    TG_TABLE_NAME,
    to_jsonb(NEW),
    to_jsonb(OLD),
    auth.uid(),
    (auth.jwt()->>'user_role')
  );
  RETURN COALESCE(NEW, OLD);
END
$$;

-- Enable on sensitive tables
CREATE TRIGGER audit_students
  AFTER INSERT OR UPDATE OR DELETE ON public.students
  FOR EACH ROW EXECUTE FUNCTION audit.log_changes();

CREATE TRIGGER audit_progress
  AFTER INSERT OR UPDATE OR DELETE ON public.progress_records
  FOR EACH ROW EXECUTE FUNCTION audit.log_changes();
```

**Confidence:** HIGH - Pattern from [Supabase blog](https://supabase.com/blog/postgres-audit) (official source), adapted for user tracking.

## Session Security (PHP)

From [PHP session security manual](https://www.php.net/manual/en/session.security.ini.php) and [OWASP CSRF Prevention](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html):

```php
// php.ini or runtime configuration
ini_set('session.cookie_httponly', 1);
ini_set('session.cookie_secure', 1);
ini_set('session.cookie_samesite', 'Strict');
ini_set('session.use_strict_mode', 1);
ini_set('session.use_only_cookies', 1);

// CSRF token generation
function generateCsrfToken(): string {
    if (empty($_SESSION['csrf_token'])) {
        $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
    }
    return $_SESSION['csrf_token'];
}

function validateCsrfToken(string $token): bool {
    return hash_equals($_SESSION['csrf_token'] ?? '', $token);
}
```

**Note:** Since using Supabase Auth, PHP sessions are minimal. Main session use is CSRF protection for form submissions. User auth state comes from Supabase JWT.

**Confidence:** HIGH - Standard PHP security practices from [official docs](https://www.php.net/manual/en/session.security.ini.php).

## Anti-Patterns to Avoid

### 1. Disabling RLS for "Simplicity"

**Trap:** supabase-client docs suggest disabling RLS for testing.
**Reality:** NEVER disable RLS in any environment. Create proper service role keys for admin operations.
**Instead:** Write RLS policies from day 1. Use service_role key only in PHP backend, never expose to client.

### 2. Storing Sensitive Data Unencrypted

**Trap:** "It's just a demo, I'll add encryption later"
**Reality:** Demo becomes MVP becomes production. Retrofitting encryption is painful.
**Instead:** Encrypt PII (student names, any identifying info) from day 1 with defuse/php-encryption.

### 3. JWT Claims Without Validation

**Trap:** Trusting client-provided role claims
**Reality:** Supabase JWTs are signed, but always validate server-side before privileged operations.
**Instead:**
```php
// Always verify JWT signature and expiration
$supabase->auth->getUser($jwt); // This validates the token
$role = $decodedJwt['user_role']; // Only then trust claims
```

### 4. Logging PII in Audit Logs

**Trap:** Logging full student records for debugging
**Reality:** Audit logs become PII liability if they contain actual student data.
**Instead:** Log record IDs, accessor IDs, and operation types. Never log actual PII values.

### 5. Using Official supabase-php (Stale)

**Trap:** Assuming official = better
**Reality:** [supabase/supabase-php](https://github.com/supabase-community/supabase-php) hasn't been updated since April 2023, is explicitly "not production ready"
**Instead:** Use [supabase-php/supabase-client](https://packagist.org/packages/supabase-php/supabase-client) (v1.2.0, Dec 2025)

### 6. Complex RLS Without Indexes

**Trap:** Writing multi-join RLS policies without supporting indexes
**Reality:** Every row check runs the policy. Complex policies without indexes = unusable performance.
**Instead:** Index EVERY column used in RLS policies. Wrap auth functions in SELECT.
