1. **Initialize Composer** (run `composer init --name=levelai/mvp --type=project --no-interaction` in project root, then `composer require supabase-php/supabase-client defuse/php-encryption vlucas/phpdotenv`). If Composer is not installed, install it first with `brew install composer`.

2. **Create `.env.example`** with these exact keys (no real values):
```
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_KEY=your-service-role-key
```

3. **Update `.gitignore`** — add these lines if not present: `vendor/`, `.env`, `composer.lock` (keep lock out of git for this demo project).

4. **Update `public/includes/bootstrap.php`** — keep existing BASE_PATH logic, add:
   - `require_once BASE_PATH . '/vendor/autoload.php';`
   - Load dotenv: `$dotenv = Dotenv\Dotenv::createImmutable(BASE_PATH); $dotenv->load();`
   - `session_start();` (if not already started)

5. **Create `public/includes/supabase-client.php`** — follow Pattern 2 from RESEARCH.md exactly:
   - `require_once __DIR__ . '/bootstrap.php';`
   - `use Supabase\CreateClient;`
   - Create client with `$_ENV['SUPABASE_URL']` and `$_ENV['SUPABASE_ANON_KEY']`
   - Return the client instance
   - Add a separate function `getAdminClient()` that uses `$_ENV['SUPABASE_SERVICE_KEY']` — NEVER expose this key in templates

6. **Create `public/includes/auth-middleware.php`** — follow Pattern 1 from RESEARCH.md exactly:
   - Read `sb_access_token` and `sb_refresh_token` from `$_COOKIE`
   - If no access token: store `$_SERVER['REQUEST_URI']` in `$_SESSION['redirect_after_login']`, redirect to `/login.php`, exit
   - Verify token via `$supabase->auth->getUser($access_token)`
   - On failure: attempt refresh via `$supabase->auth->refreshSession($refresh_token)`, rewrite both cookies with HttpOnly + SameSite=Lax + Secure flags, path=/
   - On refresh failure: clear cookies, redirect to `/login.php`
   - Extract role from `$user->app_metadata['role']`
   - If `$REQUIRED_ROLE` is set and doesn't match role: redirect to `/login.php?error=wrong_role`
   - Set `$CURRENT_USER` and `$CURRENT_ROLE` globals
   - **Do NOT write to audit_log here** — the audit_log table does not exist until Plan 01-02 creates the schema. Audit logging for page views will be added as an update to auth-middleware.php in Plan 01-02 after the table exists.

7. **Extend `public/router.php`** — add BEFORE the existing `$allowed` check:
   - Role-prefixed routes: regex `#^/(student|parent|teacher)/(.+)\.php$#` matching to `__DIR__ . "/$role/$page.php"`
   - Onboard role routes: regex `#^/onboard/(student|parent|teacher)/(step[1-9]|signup|consent)\.php$#` matching to `__DIR__ . "/onboard/$role/$step.php"`
   - Login/logout routes: add `'login'` and `'logout'` to the `$allowed` array

8. **Update `public/includes/head.php`** — add `data-role` attribute to the body tag:
   - `<body data-role="<?= htmlspecialchars($CURRENT_ROLE ?? 'guest') ?>">`
   - This enables per-role accent colors from CSS (Plan 03 implements the CSS)
